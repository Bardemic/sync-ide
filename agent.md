A provider is a file in src/providers/ that exports an object with name, aliases, description, and an open(workspaceRoot) function, registered in src/providers/index.js. Its job is to get the given absolute path in front of the target app without modifying the target app's source. Try the cleanest integration first and stop there: a documented CLI shim like zed, cursor, code, or codex app (prefer open -b with the bundle id over open -a by name when PATH shims might be hijacked); then plain open -a AppName path when Info.plist declares a folder document type; then a deep link URL scheme, which you find by reading CFBundleURLTypes and grepping the main process or asar bundle for open-url handlers (OpenCode and Superset both work this way); then a local HTTP or IPC API if the running app exposes one with a token you can read from disk (T3 Code does this); and only as a last resort a direct state file write, which means matching the app's exact SQLite or JSON schema from its bundled ORM definitions and running its full flow, not a partial one.

Core rules: never launch work the user didn't ask for, so if the only deep link available starts an agent session or creates a worktree, don't ship the provider and file an upstream request instead; don't try to solve caching problems with AppleScript keystrokes, remote debugging tricks, signals, or quit and relaunch; providers should be detached launches that return in under a second; guard everything with a darwin check and throw clearly on other platforms until someone adds cross-platform support. Before submitting, smoke test with npm link and ide <name> in a throwaway git repo, run ide list to confirm registration, update the README's provider list after.
